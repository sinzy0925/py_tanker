# TradingView MCP に関するやり取りまとめ

作成日: 2026-04-13  
対象: このチャット内での TradingView / MCP / 売買判定ロジックに関する相談内容

---

## 1. 相談の背景

ユーザーは、TradingView 上で共有されたリストの取り込み方法や、MCP（Model Context Protocol）を使った銘柄判定の自動化可否について確認した。  
また、既存の売買ロジック（陰線翌日の高値突破＋BB条件）を AI に判定させる運用の実現性、コスト最適化の方針も議論した。

---

## 2. TradingView 共有リスト取り込みに関する結論

### 2.1 共有URLからの取り込み

- 共有リストURLを開いた画面の `コピーを作成` は、実質的に「新規ウォッチリスト作成」に近い動作。
- Basicプランでは「複数ウォッチリスト」制限により、アップグレード案内が表示されるケースがある。

### 2.2 Basicプランでの現実解

- `コピーを作成` ではなく、次の方法が現実的:
  - 共有側からリストをエクスポート（TXT/CSV）
  - 自分側のウォッチリストで `Import list...`（リストをインポート）を使う
- 形式要件（公式案内）:
  - `.txt`
  - 取引所プレフィックス付きシンボル
  - カンマ区切り

### 2.3 補足

- 右パネルの `+` は1銘柄追加向けであり、大量銘柄の一括取り込みには不向き。
- 共有者が共有をOFFにするとリンク利用が難しくなるため、必要ならファイル受け渡しを依頼するのが確実。

---

## 3. 「日足で水平線上抜けで買い」案への所見

### 3.1 妥当性

- 「レジスタンス（水平線）上抜けで買い」は一般的なブレイクアウト発想で、ロジックとして不自然ではない。

### 3.2 注意点

- ヒゲ抜けと終値抜けを区別する（終値確認の方がダマシを減らしやすい）。
- 出来高の裏付けを併用すると判断の再現性が上がる。
- 損切りライン・利確ライン・サイズ管理をセットで定義する。

---

## 4. 既存ロジック（陰線翌日条件）に対する改善方針

ユーザー提示ロジック:

- 陰線の翌日（日足）が前日の陰線高値を超える
- かつ BB -1σ 内部に入る
- これを買いサインとしているが、直後の下振れが多い

### 4.1 提案された改善

- サイン即買いではなく「サイン後の確認条件」を追加する二段階判定にする。
- 例:
  - 終値で再度サイン高値を上回る
  - BB -1σ の上を維持
  - 出来高条件
  - 短期移動平均の向き
  - 直近安値割れの有無

### 4.2 実装イメージ

- スコア制（例: 5条件中3点以上で買い）
- 分割エントリー（初回50%、継続確認で追加）
- 時間切れルール（数日内に条件未達なら無効化）

## 5. 窓空き（ギャップ）判定は可能か

結論: 可能。  
MCPが直接「窓」と命名しなくても、OHLCから機械判定できる。

- ギャップアップ: `Low[t] > High[t-1]`
- ギャップダウン: `High[t] < Low[t-1]`

運用上は、最小ギャップ幅（%）や窓埋め判定を追加してノイズを減らす。

---

## 6. MCPに銘柄を渡した時に何が返るか

一般に返るのは、以下のような構造化データ:

- OHLC、出来高、騰落率
- テクニカル指標（RSI, MACD, Bollinger など）
- スクリーナー結果
- 銘柄メタ情報（取引所、タイムフレーム等）

重要点:

- 「買い/待ち」最終判定はMCPが固定で返すとは限らない。
- 返却データを使って、アプリ側で売買判定ロジックを実装する方が安定。

---

## 7. 「MCPからデータ取得にAIは必須か」

結論: 必須ではない。

- データ取得自体は API 呼び出しであり、通常コードだけで可能。
- AIは「曖昧判断」「説明生成」「ニュース要約」などに限定するとコスト効率が良い。

推奨分離:

- アプリ: 定型判定（if-then）
- AI: 例外判断・説明・優先順位づけ

## 9. コスト最適化に関する最終方針

この相談での最適方針は以下:

1. MCPでデータ取得
2. アプリで固定ロジック判定（買い/待ち）
3. 条件ヒット時のみAIを呼ぶ
4. AIには人間向け説明や例外判断を担当させる

これにより、AIトークン消費を抑えつつ、売買判定の再現性を確保できる。

---

## 10. 実装に向けた次アクション（提案）

- 売買ロジックを `if-then` で確定する（入力・判定・除外条件・損切り条件）。
- MCPレスポンスを受ける標準JSON形式を決める。
- 判定結果の出力形式を固定する（`buy|wait|invalid` + 根拠 + リスク）。
- AI呼び出し条件を限定する（例: スコア境界域のときだけ）。

## 11. 直近やり取りの追記（高コスト許容だが無駄AIは避ける方針）

ユーザー意図:

- 「最初はコスト倒れでも構わない。勝つための情報収集を優先したい」
- 「ただし、MCPサーバーから単純にデータ取得するだけの処理にAIを使う無駄は避けたい」

この意図に対する整理:

- 方針は合理的で、実務上も推奨される。
- 具体的には、処理を3層分離するのが最適:
  1. **MCPデータ取得層（非AI）**  
     API呼び出し・定期収集・保存を担当
  2. **ルール判定層（非AI）**  
     if-then/数式で `buy_candidate / wait / invalid` を出力
  3. **AI分析層（高付加価値用途のみ）**  
     例外分析、失敗要因抽出、改善提案、説明文生成

運用フロー（提案）:

1. 毎日データをMCPで取得（AIなし）
2. アプリで一次判定（AIなし）
3. 候補銘柄のみAIで深掘り
4. 週次でAIに戦略レビューを実施

AIを使わない処理（固定化対象）:

- MCPツール呼び出し
- 指標計算
- バックテスト計算
- 閾値判定・スコア算出

AIを使う処理（価値が出る領域）:

- 地合い変化の解釈
- 連敗パターンの共通点抽出
- ルール改善案の生成
- 人間向け要約レポート作成

無駄AIを防ぐガードレール:

- AI呼び出し条件を明文化（例: 上位候補のみ）
- 同一データへの重複問い合わせ防止
- 入力を要約JSONに限定
- 1日あたり呼び出し上限設定

要点:

- **「データ取得・計算・一次判定は非AI」**
- **「AIは高付加価値の判断補助に限定」**

この分離により、情報収集の厚みを確保しつつ、AIコストの無駄打ちを抑制できる。
